# ADR-001: Выбор механизма перехвата изменений

**Статус:** Принято  
**Дата:** 2026-04-29

---

## Контекст

Задача: отслеживать изменения на настраиваемых моделях и полях **без изменений кода** при добавлении или удалении модели из отслеживания. Конфигурация должна управляться через UI.

---

## Рассмотренные варианты

### 1. OCA `auditlog` модуль (готовое решение)

Существует production-ready модуль от Odoo Community Association:
[github.com/OCA/server-tools/tree/16.0/auditlog](https://github.com/OCA/server-tools/tree/16.0/auditlog)

- Широко используется в индустрии
- Протестирован на десятках production-систем
- Поддерживается сообществом

**В реальном проекте** — предпочтительный выбор: взять OCA-модуль как основу и доработать.  
**В данном случае** — написано с нуля для демонстрации понимания механики и архитектурных решений.

---

### 2. Mixin + ручное наследование

Добавить mixin-класс в каждую отслеживаемую модель.

```python
class CrmLead(AuditMixin, models.Model):
    _inherit = 'crm.lead'
```

**Отклонено:** нарушает условие задания — требует изменений кода при каждом добавлении новой модели.

---

### 3. Monkey-patch `BaseModel.write()`

Один глобальный патч на базовый класс всех моделей Odoo.

**Отклонено:** патчит все 500+ моделей Odoo, включая системные. Неприемлемый performance overhead и сложность отладки.

---

### 4. PostgreSQL triggers

Триггеры на уровне базы данных, перехватывающие `UPDATE`.

**Отклонено:**
- Не видит `uid` пользователя Odoo (только DB-пользователь)
- Сложно отображать в UI без дополнительного слоя
- Конфигурация через UI становится нетривиальной

---

### 5. `_register_hook()` + динамический патч (выбрано)

При старте Odoo `_register_hook` читает активные правила из `audit.rule` и динамически оборачивает `write()` только у нужных моделей.

```
_register_hook() → ищет активные audit.rule
    → для каждой модели: _patch_write(Model)
    → Model.write = audited_write (closure над original_write)
```

**Принято по следующим причинам:**
- Конфигурация через UI — код не меняется при добавлении/удалении модели
- Патчатся только модели с активными правилами
- При создании/изменении правила патч применяется сразу, без рестарта
- Идиоматичный Odoo-подход, используемый в ядре и OCA-модулях

---

## Последствия

- При **деактивации** правила патч остаётся в памяти до рестарта сервера. Это приемлемо: `audited_write` проверяет активные правила при каждом вызове и просто пропускает логирование если правило неактивно.
- При **рестарте** Odoo `_register_hook` применяет только актуальные активные правила.
